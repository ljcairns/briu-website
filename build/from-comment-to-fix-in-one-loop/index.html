From Comment to Fix in One Loop

How we wired a customer feedback notification into an agent that reads the conversation, updates the live product, and emails the client — without us re-learning the account.

The Real Cost of a Customer Comment

We run chatbot deployments for clients through Subsights. A customer leaves a comment on one of their conversations — the opener is too narrow, a product line isn't triggering, the staff contact listed is someone who left last month. The fix itself is five minutes.

But the fix isn't what takes the time.

The context reload is what kills you. Log into the Subsights dashboard. Find the right workspace — which client account was this again? Find the right chatbot. There might be three per workspace. Navigate through accordion menus to the prompt editor. Re-read the configuration to remember what you set up two months ago. Parse the customer's comment to understand what they're actually asking for. Make the change in the right textarea. Test it. Then figure out who to email — was it the ops lead or the founder? Is that email address even current? Write the email. Send it. Mark the comment as handled so nobody else picks it up.

That's not five minutes. That's twenty to thirty. And for a small team, that friction means comments sit for days. Not because the work is hard. Because starting the work is hard.

The Loop We Built

Subsights posts a webhook to a Discord channel whenever a customer comments on a conversation. A monitoring script checks that channel every 30 minutes. Here's exactly what happens when it finds one:

Detect & Dedup

Script pulls new comments from the Discord webhook channel. Cross-references a SQLite table of already-processed comments. New ones get queued.

Read Conversation

Agent opens the actual customer conversation via browser automation. Reads the full thread — what the customer said, what the chatbot said, where it went wrong.

Read Live Config

Navigates the Subsights dashboard — workspace switching, accordion menus, prompt editor. Reads the current chatbot configuration so it knows exactly what to change.

Make Changes

Edits the prompts in the platform's textareas. Broader opener, new product triggers, updated staff contact — whatever the feedback requires.

Verify

Saves, reloads the configuration page, confirms the changes persisted. No "I think I saved it" — the agent checks.

Client Email

Queries the client_contacts database for the right person at that workspace. Drafts a specific email explaining what changed and why. Sends on human approval.

Close the Loop

Marks the comment as processed in SQLite. Posts a status update back to the Discord thread. The next comment starts a fresh cycle.

The whole cycle runs from a single heartbeat cron. The agent already knows the workspace structure, the contact database, the navigation pattern for each platform section. It doesn't need to re-learn the account because the account context lives in queryable database tables — not in someone's head.

What Broke and What We Documented

It didn't work cleanly the first time. Or the second. Every failure became a permanent upgrade to the system.

Wrong chatbot name

Workspace had three chatbots with similar names. Agent picked the wrong one and edited the wrong prompt.

Fix: Built a client_chatbots database table mapping workspace IDs to the correct chatbot. Replaced hardcoded chatbot references across 6 files with one queryable source.

Accordion menus and nested UI

The Subsights prompt editor has collapsible sections. Agent couldn't find the right textarea because it was hidden inside a collapsed panel.

Fix: Browser navigation guide updated with explicit expand-then-edit sequences for each section type. Different DOM selectors per workspace version.

Stale contact information

Commenter's email belonged to a former employee who had left the company. Agent would have emailed someone who no longer works there.

Fix: Agent now cross-references the client_contacts table and validates against Gmail history. Found the real current contact before sending.

Each failure became a permanent system upgrade. The client_chatbots table, the client_contacts database, the browser navigation guide, the comment processing script — they all get more reliable with every cycle. The system learns from its mistakes once. A human re-learns them every time they context-switch back into an account.

What It Feels Like Now

The heartbeat cron runs every 30 minutes. Comments come in. The loop runs. Changes go live. The client gets an email explaining exactly what was updated and why. A status update posts back to the Discord thread.

The account context that used to live in someone's head — which chatbot, which workspace, who's the contact, what did we configure last time — now lives in shared database tables (client_chatbots, client_contacts) and a browser navigation guide. The agent queries them every time. It doesn't have good days and bad days. It doesn't forget that the contact changed last month. It doesn't dread the small task that takes longer to start than to finish.

The human role shifted from "do the whole thing" to "review the email before it sends." That's the right division. The agent handles the mechanical recall and execution. The human handles the judgment call on tone and correctness.

The Knowledge Coherence Problem

Before the loop, operational context was scattered. Chatbot IDs were hardcoded across 6 different files. Client contacts lived in email threads. Navigation patterns lived in muscle memory. Every time the agent (or a human) needed to service an account, they had to reconstruct this context from scratch.

The loop forced us to consolidate. We built shared database tables — client_chatbots maps every workspace to its correct chatbot ID. client_contacts stores the current point of contact for each account, validated against real email history. The browser navigation guide documents the exact click sequence for each workspace type.

This is the real output of the project. Not just the automation — the structured knowledge that makes the automation possible. The loop runs because the context is encoded. And once it's encoded, it's available to every other process that touches that account. The comment loop was the forcing function. The knowledge system is the lasting asset.

Why This Matters Beyond Us

Every service business has this pattern. Small customer requests that are easy to do but painful to start. The fix is never the bottleneck. The context reload is.

Think about the tasks your team dreads — not because they're hard, but because they require re-acquainting with an account, a platform, a relationship. Those are the tasks that sit in a queue for days. Those are the tasks that make customers feel forgotten.

The pattern is straightforward:

  • Encode the context. Contacts, credentials, navigation patterns, preferences, configuration history. Get it out of people's heads and into structured reference documents.
  • Wire the trigger. Notification, email, form submission, webhook — whatever event signals "a customer needs something."
  • Connect trigger to resolution. The agent reads the trigger, consults the context, executes the fix, communicates the result.

The loop eliminates the context switch. The human stays in the loop for judgment. The customer gets a response in minutes instead of days.

That's the service we deploy for clients. Not the agent itself — the elimination of the context reload that makes small tasks feel big.
