How We Secure AI Agent Systems

Continuous adversarial testing against a defense-in-depth architecture. This is the methodology we use for our own systems and deploy for clients.

The problem with AI security theater

Most AI security is checkbox compliance: “we use encryption” (everyone does), “we follow SOC 2” (that’s about process, not agent behavior), “our models are aligned” (alignment is a property of training, not a runtime guarantee).

None of that addresses the actual threat: an autonomous AI agent connected to your email, CRM, and business tools can be manipulated through the content it processes. An email subject line. A calendar invite. A form submission. A webhook payload. All of it enters the model’s context window alongside its instructions.

The question isn’t whether your model is aligned. The question is what happens when an attacker puts instructions inside an email and your agent reads it.

Our approach: defense in depth, tested continuously

Principle 1: Every layer assumes the others failed

We don’t build one strong wall. We build ten independent layers, each designed to contain a breach that got past the ones before it.

  1. Input sanitizer — 140+ patterns, 7 detection stages, 20+ languages. Catches known injection patterns before the model sees them.
  2. Auto-sanitizer hook — plugin-level enforcement. Every tool result over 50 characters is scanned. The agent cannot opt out.
  3. Structural framing — all external content wrapped in XML tags that mark it as data, not instructions. Works even when regex fails.
  4. Sandbox isolation — external content processed by a restricted agent (read-only, no tools) with structured JSON output. Re-sanitized by a deterministic pipeline (not AI) before the operational agent sees it.
  5. Canary tokens — daily-rotating fake credentials planted in the agent’s context. If extracted, the leak is detected and blocked.
  6. Write protection — critical config files locked from all non-operator agents. Blocks direct and indirect writes.
  7. Exec guardrails — deny-by-default command allowlist. Blocks code execution, network exfiltration, and file writes for non-main agents.
  8. Behavioral watchdog — anomaly detection with three-tier kill switch. Critical CVEs trigger automatic lockdown.
  9. HMAC task signing — cryptographic verification of all inter-agent task manifests with nonce replay protection and 60-second expiry windows.
  10. Pipeline exec parsing — structural command parsing that validates every segment in pipes and chains against the allowlist independently.

Principle 2: Pattern matching has a ceiling

After 290+ attack payloads across 8 red team rounds, we know exactly where regex-based detection stops working: pure semantic metaphor, novel encoding schemes, sub-threshold payloads, and composition attacks where each piece is clean individually. We don’t pretend to catch these with patterns. We catch them with structural defenses (layers 3–10) that don’t depend on recognizing the attack.

Principle 3: Infrastructure enforces, prompts suggest

Every security control is enforced at the infrastructure level, not the prompt level. An agent’s permissions are set by tool policy, not by instructions it could be manipulated into ignoring. The difference: a prompt-level instruction is a suggestion to a language model. An infrastructure-level control is a guarantee.

Architectural audits

Payload testing only exercises layer 1. The real vulnerabilities are in assumptions: plugin return fields the runtime silently ignores, execution order bugs that scan content after it’s been truncated, trust boundaries where any agent can write to state that another agent follows as instructions.

We run 4 parallel audit tracks:

Plugin interaction audit

Hook execution order, cross-plugin data flow, return field contracts with the runtime.

Trust boundary audit

Which agents write to which state tables, what the main agent consumes as trusted.

External data flow audit

Trace every path from attacker-controlled input (email, calendar, webhook, form) through sanitization to model context.

Permission and filesystem audit

Unix permissions, symlink validation, argument injection in CLI wrappers.

Multi-agent red team

For systems with multiple agents (personal + team, VPS + Mac, multi-model orchestration), we test the boundaries between them:

  • HMAC task signing — replay attacks, timing side channels, key extraction
  • Transport integrity — message forgery, reordering, replay via Discord
  • Cross-agent escalation — can a restricted agent compromise a privileged one?
  • Compaction trust loss — security context stripped during context compression

What this means for clients

Every agent system we deploy gets this methodology applied:

  • Pre-deployment: architectural review, permission scoping, defense layer configuration, initial red team round
  • At deployment: full payload testing + architectural audit against the client’s specific integration surface
  • Ongoing: continuous red teaming on retainer. New plugins, new tools, new data sources all get tested before they go live
  • Transparent: clients get the full report. Every finding, every severity rating, every fix. No black boxes.

    • We run frontier models (Sonnet 4.6, Opus 4.6) for stronger baseline instruction resistance, but we never rely on model behavior alone. Models are one layer. Infrastructure is the other nine.

    The compaction problem (and why most agent security ignores it)

    One finding from our red team work stands apart because it’s a fundamental property of how LLM sessions work, not something specific to any framework.

    When a session gets long, the runtime compacts it: summarizes old messages to free context window. During compaction, structural framing tags get stripped. Injection payloads that were safely framed become unframed text in the summary.

    An attacker who can get a payload into the session and wait for compaction gets a free de-framing pass. The payload enters wrapped. It exits naked.

    Our mitigations: re-sanitize summaries after compaction, track “contains untrusted content” metadata through the pipeline, inject explicit rules about compacted content into every turn. Most agent deployments we’ve reviewed do none of these.
