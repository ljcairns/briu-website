Security Architecture

Three dimensions. No single score.

An AI agent that handles email, CRM, and financial data needs more than a single defense. Here is how this stack is structured, tested, and operated.

Architecture
10 Defense Layers

Each assumes the others failed.

  1. Input sanitizer — 140+ patterns, 7 detection stages, 20+ languages
  2. Auto-sanitizer hook — enforced scanning on every tool result
  3. Canary tokens — daily-rotating fake credentials as tripwires
  4. Write protection — critical files locked from non-operator agents
  5. Exec guardrails — blocks code execution & network exfiltration
  6. Kill switch — automated lockdown on critical CVEs
  7. Sandbox isolation — external content via sandboxed agent
  8. Structural framing — all external data tagged as untrusted
  9. HMAC task signing — cryptographic verification of inter-agent task manifests with nonce replay protection
  10. Pipeline exec parsing — structural command parsing validates every pipe/chain segment against the allowlist

Operating Posture
Human approval
for external actions
  • Every external action requires explicit approval before it runs
  • Per-agent budget ceiling — hard stop at threshold, no overrides
  • Permission scope locked at deployment, not adjustable at runtime
  • All inbound content sanitized before reaching any model
  • Anomaly detection + automatic lockdown on threshold breach

We use frontier models for stronger instruction resistance, but we do not rely on model behavior alone. These controls are how we contain failures when they happen.
