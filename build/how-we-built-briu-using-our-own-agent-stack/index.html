Building Briu with Agents

From first agent skill to company launch. Exact toolchain, costs, and what the agent cannot do. Specific enough that you could replicate it.

Why We Started Building

Jason Calacanis on the All-In Podcast: "Every week 5 to 10% of the work we're doing inside of our venture firm is being moved over to OpenClaw." That number is cumulative and permanent. Move 5% per week, and in 20 weeks you have automated everything you currently do by hand.

The agent

Before Briu existed, there was an agent built through Subsights, our previous business. It started simple: processing inbound email, triaging messages, drafting responses for review. Then it expanded. Calendar management. Daily reporting. CRM updates through HubSpot. Prospecting through Apollo. Research through web access.

Each new skill followed the same process: build, test in isolation, connect to live systems with approval gates, monitor output, gradually reduce oversight as trust was earned. The agent never takes an external action without human approval. It can draft an email — it cannot send one. It can propose a calendar change — it cannot confirm one.

Today, I do not go into my email. I do not manage my calendar. Those things are done for me, reviewed by me, and executed on my approval.

The toolchain

Claude Max Plan

$200/month. Primary reasoning interface for strategic work — positioning, copy, analysis, planning.

Claude Code

Agent-powered development in the terminal. The entire Briu website was built this way — HTML, CSS, JavaScript, responsive layouts, interactive components.

OpenClaw Agent

The orchestration layer. Manages the agent's skills, system connections, daily routines, and task execution. Email, calendar, CRM, web access, reporting — all structured as skills with defined inputs, outputs, and approval gates.

Security

When your agent is connected to your email, CRM, calendar, and customer data, the security model is the foundation. Every skill is built in-house. Every system connection has defined scopes and permissions. We built an internal security agent whose sole job is to audit the primary agent's access patterns and flag anomalies.

The architecture is multi-agent: the primary agent handles high-trust tasks on the most capable model, isolated sub-agents handle lower-risk work on cost-effective options. Frontier-tier models for anything touching external data or customer information. Cheaper models for internal summarization, formatting, reporting.

Building the brand with the stack

Once the agent system was stable, we used the same stack to build the company. Briu was built with the tooling from day one.

Name and domain. Claude explored naming directions across multiple languages. Hundreds of candidates, each evaluated against availability, trademark risk, and brand fit. We built an OpenClaw skill mid-process to automate domain search across 50+ candidates. "Briu" emerged from Catalan — a word meaning energy, push, courage. A human made the final call.

Visual identity. 400+ Midjourney generations. Each round directed by founder taste, informed by the previous generation. The Catalan origin pointed toward Gaudi, and that aesthetic — pointed arches, mosaic tilework, warm gold — became the visual starting point. The logo was assembled in Canva and the brand system carried directly into the website build.

Website. Built through Claude Code in the terminal. Describe what a section should accomplish, the agent writes the code, review in the browser, adjust in real time. The contact modal, service tiers, responsive navigation, economics charts — all built this way.

Content. Site copy went through dozens of iterations. Claude on the web for strategic thinking. Claude Code for implementation. Every quote sourced with episode-level attribution.

What the agent cannot do

The agent produced technically excellent work that completely missed the point. A services page that positioned Briu like an enterprise vendor. A homepage that felt cold. A pricing section that sent the wrong signal about who we serve.

The agent cannot feel the gap between "correct" and "right." The goal is not full automation. The goal is a system where the human spends time on judgment instead of production.

The pattern we deploy for clients

1. Identify the workflow

Find the repeatable process that consumes disproportionate time relative to the judgment it requires.

2. Structure and build

Break it into discrete steps with clear inputs, outputs, and review points. Build the skills. Connect the systems. Set up approval gates.

3. Deploy, monitor, expand

Start narrow. Human at every decision gate. Expand scope after trust is earned. Then move to the next 5%.

Same approach whether building a first email automation or a full multi-agent system. The tools change. The pattern holds.

Services now, systems later

Being services-led is not temporary. Every deployment generates patterns. Over time, those patterns become standardized systems and eventually software. But that only works if the implementation experience is real.
