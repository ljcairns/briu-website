How We Built Briu Using Our Own Agent Stack

We built a production agent, connected it to our real systems, and used it to launch an entire company — from brand creation to a live website. This is exactly how we did it, what it cost, and what we learned. Specific enough that you could replicate it yourself.

Why We Started Building

It started with a simple realization. Jason Calacanis described it on the All-In Podcast: "Every week 5 to 10% of the work we're doing inside of our venture firm is being moved over to OpenClaw." That number is not hypothetical. It is cumulative and permanent. Once a workflow is automated, it stays automated. Do the math: if you can move even 5% of your workload to an agent every week, in 20 weeks you have automated everything you currently do by hand.

Once that clicked, the question stopped being "should we use agents" and became "how fast can we get this running." So we started building.

Meet the agent

Before Briu existed as a company, there was an agent. We built our first production agent through Subsights, our previous business. The goal was not a demo or a proof of concept. The goal was a system that could handle real daily operations — and that I could trust with my actual inbox, calendar, and business tools.

The agent — we call him Jannik — started simple: processing inbound email, triaging messages, drafting responses for review. Then it expanded. Calendar management. Daily reporting across all connected systems. Creating customer demos through the Subsights web app. CRM updates through HubSpot. Prospecting workflows through Apollo. Research and competitive analysis through web access.

Each new skill went through the same process: build it, test it in isolation, connect it to live systems with approval gates, monitor the output, then gradually reduce oversight as trust was earned. The critical principle was that the agent never takes an external action without human approval. It can draft an email — it cannot send one. It can propose a calendar change — it cannot confirm one. Every outbound action goes through a gate.

The goal was to build a production-grade agent I could use to run my life. Not a toy. Not a chatbot. A system that handles the operational load so I can focus on judgment, strategy, and the work that actually requires a human.

Today, I do not go into my email. I do not manage my calendar. Those things are done for me, reviewed by me, and executed on my approval. And honestly — it is great. The cognitive load reduction alone is worth the entire investment. But the real value is what it frees up: time and attention for the work that moves the business forward.

The toolchain: exactly what we use

We want to be specific here because vague descriptions of "AI-powered workflows" are useless if you are trying to replicate this. Here is exactly what we run:

Claude Max Plan

$200/month. This is the primary reasoning interface for strategic work — positioning, copy, analysis, planning. The web app is where high-level thinking happens: iterating on brand direction, stress-testing market positioning, drafting and refining long-form content.

Claude Code

Agent-powered development in the terminal. This is how the entire Briu website was built — HTML, CSS, JavaScript, responsive layouts, interactive components, form logic. Iterative conversation with the model, reviewing changes in the browser, shipping in real time.

OpenClaw Agent

The orchestration layer. OpenClaw manages Jannik's skills, system connections, daily routines, and task execution. Email, calendar, CRM, web access, reporting — all structured as agent skills with defined inputs, outputs, and approval gates.

These three tools work together but serve different functions. Claude on the web is for thinking. Claude Code is for building. OpenClaw is for operating. The distinction matters because it shapes how you structure work: strategic decisions happen in conversation, code happens in the terminal, and recurring operations happen through the agent framework.

Security: why we built everything in-house

This is the part most people skip, and it is the part that matters most. When your agent is connected to your email, your CRM, your calendar, and your customer data, the security model is not optional. It is the foundation.

We do not use any outside skills or third-party agent plugins. Every skill is built in-house. Every system connection is implemented directly with defined scopes and permissions. The agent has its own security architecture — daily security checks, trend monitoring, vulnerability scanning across all connected systems. We built an internal security agent whose sole job is to audit the primary agent's access patterns, flag anomalies, and verify that data handling meets our standards.

For systems handling sensitive data, you need to be running frontier-tier models. It gets costly, but that is what you have to do to be safe today. Cheaper models are fine for internal tasks that do not carry prompt injection risk — summarization, formatting, internal reporting. But anything touching external data or customer information needs the strongest model available.

The practical architecture is a multi-agent system where the primary agent handles high-trust tasks on the most capable model, and isolated sub-agents handle lower-risk internal work on more cost-effective options. This lets you optimize spend without compromising security on the tasks that matter. We identified all major risk areas — prompt injection, data exfiltration, scope creep, unauthorized actions — and have systems in place for each one. As Chamath put it: "If I'm GEICO, do I want to have all of my actuaries using all of our proprietary, private, and confidential data in an open instance of an LLM? The answer is no."

Building the Briu brand with the stack

Once the agent system was stable and running daily operations, the next question was obvious: use the same stack to build the next business. Briu did not start as a company and then adopt AI tooling. It was built with the tooling from day one.

Name and domain. We used Claude on the web to explore naming directions — geographic roots, Gaelic language patterns, abstract concepts, industry terms. The agent generated hundreds of candidates. Each was evaluated against availability, trademark risk, pronunciation, and brand fit. The search space was wider than any manual process would allow. "Briu" emerged from an exploration of Scottish Gaelic — it means "meaning" or "significance." A human made the final call, but the agent did the legwork that made an informed call possible.

Visual identity. We iterated through hundreds of images in Midjourney to explore brand directions — color palettes, textures, moods, typographic styles. This was not random generation. Each round was informed by the previous one, with the founder directing the aesthetic and the tools expanding the range of options. The logo was assembled in Canva, refined through several rounds, and the final brand system — colors, fonts, spacing, tone — was codified and carried directly into the website build.

Website. The entire site was built through Claude Code in the terminal. Not generated once and handed off — built iteratively through conversation. The founder described what a section should accomplish, the agent wrote the code, the founder reviewed it in the browser, and adjustments happened in real time. The contact modal, the service tier layouts, the responsive navigation, the quote styling, the economics charts — all built this way. When something looked wrong, the fix was a sentence, not a ticket. The speed was genuinely shocking. Sections that would take a developer hours to implement were live in minutes.

Content and positioning. The site copy went through dozens of iterations. Claude on the web handled the strategic thinking — what claims can we credibly make, what tone separates us from generic AI consultancies, what does the reader actually need to hear. Claude Code handled the implementation — taking refined copy and building it into page structure with proper styling and layout. Every quote on the site was sourced with episode-level attribution and verified against its original context.

What the agent cannot do

There were moments where the agent produced technically excellent work that completely missed the point. A beautifully structured services page that positioned Briu like an enterprise vendor. A homepage that was impressive but felt cold. A pricing section that was logical but sent the wrong signal about who we serve.

The agent cannot feel the gap between "correct" and "right." It does not know when a sentence sounds polished but hollow. It does not know when a site feels premium versus merely expensive-looking. It does not know which direction is worth staking a business on. Those are founder decisions. Peter Steinberger, the creator of OpenClaw, put it well: "People who try to automate the whole thing... it misses style, love, that human touch. I don't think you can automate that away so quickly."

The goal is not full automation. The goal is a system where the human spends their time on judgment instead of production.

Why your employees are your biggest advantage

One of the clearest insights from watching teams adopt agents: the people who already understand your business are the ones who get the most out of this. David Sacks made the point directly: "A key skill of employees is going to be the ability to structure work for themselves and their AI agents."

The playbook is straightforward. Get your most technical people onto agents first. Teach them how to build workflows, how to structure tasks, how to set review points. Then keep going — expand to the next tier, and the next. The tools are getting better fast enough that eventually even the least technical person on your team can be an agent orchestrator, effectively becoming a 10x or 100x version of themselves.

The reason you keep your employees around is not that they can type faster or process more spreadsheets. It is that they have institutional knowledge — the nuanced, hard-to-capture understanding of your systems, your customers, your edge cases, your culture. That human judgment is what makes agent systems actually work instead of just producing plausible-looking output. You do not replace your people. You power them up.

Jason Calacanis on the emerging role: "Every startup I know is hiring for this position, which is somebody who knows how to build and manage agents. There is no job rec for this yet or a title." That role is what Briu helps create inside your organization.

The pattern we deploy for clients

Building Briu this way produced the operating playbook we now use in every engagement:

1. Identify the workflow

Find the repeatable process that consumes disproportionate time relative to the judgment it requires. Email triage. CRM updates. Report generation. Prospecting. Scheduling.

2. Structure and build

Break it into discrete steps with clear inputs, outputs, and review points. Build the agent skills. Connect the systems. Set up approval gates. Agents need boundaries, not freedom.

3. Deploy, monitor, expand

Start narrow. Let the agent handle production work. Keep a human at every decision gate. Expand scope only after trust is earned. Then move to the next 5%.

This is the same approach whether we are building a client's first email automation or deploying a full multi-agent system with security monitoring. The tools change. The pattern holds. And it gets easier every time because the pattern library grows with each deployment.

Services now, systems later

Being services-led is not a temporary phase to explain away. It is the mechanism by which real product knowledge is earned. Every client engagement, every internal workflow, every deployment generates patterns. Over time, those patterns become more standardized systems, more reusable tooling, and eventually software. But that only works if the underlying implementation experience is real.

That is why Build exists. Not to publish generic takes on AI, but to document the operating layer underneath the business and show what these systems look like when they are used for actual work. If you read this and thought "I could do this" — you are right. And if you want help getting there faster, that is exactly what we do.
