What It Does

Every weekday at 6:54 AM, a shell script fires. It queries 10 separate database tables, checks 3 email accounts (9 queries total), pulls CRM pipeline data from HubSpot, fetches 2 calendars, gets cost data from the gateway API, and pulls a daily report from Subsights. All of it gets assembled into a single JSON snapshot file.

At 7:04 AM, the agent reads that snapshot and composes a structured briefing. Before writing a single line, it runs a reconciliation pass: cross-checking email subjects against state-db action records, flagging anything that doesn't match. Then it writes 9–11 sections and posts to Discord, split across 3–4 messages due to Discord's 2K character limit.

Total time from trigger to delivery: about 2 minutes. A human doing this same work across the same sources would need 30–45 minutes. Every morning.

The Three-Stage Pipeline

The briefing isn't one script. It's three stages, each with a specific job and failure mode.

Pre-Fetch

235-line shell script. Runs 10 minutes before composition. Queries state-db, 3 email accounts, HubSpot CRM, 2 calendars, cost API, Subsights. Assembles everything into one JSON file with atomic rename for safety.

6:54 AM · ~30 seconds

Preflight

31-line validation script. Checks that the snapshot is fresh (<15 min old), all data sources are populated, and cost data is current. Blocks composition if anything is stale.

7:03 AM · ~2 seconds

Composition

230-line prompt. Sonnet 4.6 reads the snapshot, runs reconciliation, composes 10 sections with chain-format grouping. Posts to Discord #home. 7–21 turns, 165K–750K tokens.

7:04 AM · ~90 seconds

Data Sources

The pre-fetch script hits all of these before the agent runs. The agent never makes API calls during composition — it reads from a pre-assembled snapshot. This keeps token usage predictable and composition fast.

State DB 10 queries: actions, decisions, tasks, clients, contacts, notes, config, costs, security events, reconciliation log
Email 3 accounts × 3 queries each: unread count, recent subjects, flagged messages
HubSpot CRM Pipeline deals with pinned contact/deal filtering via jq
Calendar 2 accounts: today + tomorrow events
Cost API Gateway spend data: yesterday, running total, per-model
Subsights Daily report: new comments, conversation metrics

Why Pre-Fetch Matters

Without pre-fetch, the agent would need to make 20+ API calls during composition. Each call burns tokens on the request, the response, and the agent reasoning about what to do with it. A 750K-token briefing becomes a 2M-token briefing. Cost triples. Latency goes from 90 seconds to 8 minutes.

The pre-fetch pipeline also acts as a circuit breaker. If an email API is down, the snapshot still has partial data and the preflight script flags what's missing. The agent doesn't hallucinate data it couldn't fetch — it sees explicit gaps in the snapshot and reports them.

The snapshot is an atomic boundary. Everything the agent needs to compose the briefing is in one file. If the file is fresh and complete, composition succeeds. If it's stale or partial, preflight blocks it. No hidden dependencies.

What It Catches

The briefing's value isn't the summary — it's the things it surfaces that you wouldn't have noticed.

Vanishing Records

Between 6:08 AM and 7:04 AM on Feb 28, actions #9 and #10 disappeared from state-db. Briefing caught the delta: “Verify both HubSpot entries were actually created — or confirm actions were intentionally closed.” They hadn't been created.

18-Month Stale Deal

Visit Sun Valley HubSpot deal (stage: presentationscheduled, last touched Aug 2024) appeared under DRIFT in every briefing. Cross-referenced against state-db pipeline, found no matching entry. Flagged daily until resolved.

12-Hour Deadline Surface

Google AI Studio deprecation email (March 9 deadline) arrived Feb 26. Surfaced as Priority #3 by 7:04 AM Feb 27: “decide: migrate or drop the API.” Turned a buried email into a binary decision.

Cross-Source Linking

Anthropic invoice in personal Gmail (Feb 26, 9:29 AM) cross-referenced with pre-scheduled “Do all taxes” calendar event (Mar 3, 8 AM). Surfaced as single linked item. Neither obvious alone.

Self-Recovery

Feb 28, 6:08 AM cron session timed out. Briefing re-ran at 7:04 AM with warning header, reported snapshot failure, still delivered complete briefing. Zero manual intervention.

Client State Mismatch

Dylan's Tours showed “active” in state-db but “churned Feb 2026” in MEMORY.md. Briefing flagged under DRIFT: “Update state-db to churned.”

Phishing Detection

Fake USPTO trademark notice from tsdrusptonotification.com flagged during a routine email scan. The agent recognized the spoofed domain and surfaced it as a security alert instead of passing it through as a legitimate action item.

What Makes It Non-Trivial

A daily summary sounds simple. Read some email, check the calendar, post to Slack. The hard parts are the parts you don't see until you try to build it:

  • Cross-referencing across 6+ data sources. An email about a deal needs to match the CRM record and the calendar meeting and the state-db action. The briefing chains these together instead of listing them as separate items.
  • Reconciliation. The agent checks whether email threads match state-db records before composing. If someone replied to a thread the agent doesn't know about, it flags it as an orphan rather than ignoring it.
  • Self-referential cost tracking. The briefing includes a cost report for the system that generates the briefing. The cost section tells you what you're paying for the thing you're reading.
  • Drift detection. When the system's model of reality diverges from actual state — a deal moved in CRM but the state-db still shows the old stage — the briefing surfaces it explicitly.

The pre-fetch architecture means the agent doesn't waste tokens discovering what's happening. It already knows. Composition is just formatting and judgment — which items to chain together, which to flag, which to surface at the top.
