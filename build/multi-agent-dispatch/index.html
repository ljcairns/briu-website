How It Started

The first version was one agent on one VPS. A cron job woke it up, it read a task queue from a SQLite database, ran the task, wrote the result back. Simple. It worked for weeks.

Then the task list grew. Some tasks needed browser automation. Some needed local file access. Some needed GPU inference. One machine couldn't do all three, and spinning up a bigger VPS wasn't the answer — the tasks had fundamentally different runtime requirements.

The question shifted from "how do we run tasks" to "how do we route them."

The Evolution

Mac Backend + HMAC Signing

Added a Mac Mini for tasks requiring local filesystem access and browser automation. The VPS dispatches tasks to the Mac over HTTPS. Every request signed with HMAC-SHA256 — timestamp in the header, body in the signature. Replay window: 60 seconds. No shared secrets in transit.

Week 5
03

Three-Backend Dispatch

Third backend for heavy inference and batch processing. Dispatcher on the VPS reads the task queue, checks task metadata, routes to the correct backend. Each backend has its own health check and failure mode. If one goes down, its tasks queue up instead of failing.

Week 7

The Routing Problem

Three backends worked. Routing didn't. The LLM-based router couldn't reliably decide where tasks should go. Latency spiked. Misroutes caused retries. The dispatch layer was burning more tokens than the tasks themselves.

Week 8 — the wall

Mechanical Plugin Router

Replaced the LLM router with a deterministic plugin system. Task type maps to backend via a config table. No inference at the routing layer. Zero ambiguity. Dispatch went from 4–12 seconds to under 50 milliseconds.

Architecture Now

The dispatcher sits on the VPS. It reads from the task queue, looks up the task type in a plugin registry, and routes to the correct backend. No LLM in the routing path. The LLM only runs on the backend that executes the task.

The Routing Problem

What Went Wrong

The original router was an LLM call. Read the task description, decide which backend. Sounded reasonable. In practice:

Ambiguity killed it. "Summarize the latest client emails and update the CRM" touches both API calls and local context. The LLM would flip between Backend A and Backend B across identical tasks. Same prompt, different routing, depending on which tokens got sampled.

Latency compounded. A 4–12 second routing decision before a 37-second task. For batch jobs with 20 tasks, that's 80–240 seconds of pure overhead just deciding where to send work.

Cost inversion. The router was spending $0.03–0.06 per decision. For a $0.08 API task, routing cost more than execution. The dispatch layer was 40% of total spend.

The Mechanical Fix

Replaced the LLM router with a plugin registry. Each task type has a declared backend affinity in a config table. The dispatcher does a dictionary lookup, not an inference call.

Result: routing dropped from 4–12 seconds to <50ms. Zero ambiguity. Zero token cost at the routing layer. Misroutes went from ~8% to 0%.

The insight: routing is a classification problem with known categories. You don't need a language model to look up a key in a table. Save the inference budget for the actual work.

HMAC Signing

The Mac backend accepts work from the VPS dispatcher over HTTPS. Every request is signed with HMAC-SHA256. The signature covers the request body and a timestamp header. Replay window is 60 seconds — any request older than that gets rejected.

No shared secrets travel over the wire. Both sides have the key pre-provisioned. The Mac validates the signature, checks the timestamp, then executes. If the signature doesn't match or the timestamp is stale, the request is dropped with a 401. No error detail in the response — failed auth looks identical regardless of failure mode.

This is table stakes for multi-machine dispatch. Without request signing, anyone who discovers the Mac's endpoint can submit arbitrary tasks. With HMAC, the dispatcher proves identity on every request without a session, token exchange, or certificate authority.

What 47 Findings Looks Like

Running the dispatch system for three weeks produced 47 distinct operational findings. Not bugs — findings. Things the system surfaced that would have gone unnoticed in a single-machine setup.

Silent Queue Starvation

Backend C went idle for 6 hours while 11 batch tasks queued. Health check passed (process alive) but the inference endpoint had silently OOM'd. Added memory-pressure check to health probe.

Clock Drift Rejection

Mac's system clock drifted 73 seconds from VPS. HMAC replay window (60s) started rejecting valid requests. Looked like an attack. Was NTP misconfiguration. Added clock-skew monitoring to the dispatcher.

Cost Asymmetry

Backend A handled 60% of tasks but only 19% of spend. Backend C handled 15% of tasks but 47% of spend. Led to the per-backend cost tracking that now feeds the morning briefing.

Retry Amplification

A transient Mac network blip caused 3 tasks to retry. Each retry re-queued the task, which re-triggered dispatch, which re-signed the HMAC. 3 tasks became 9 dispatch events in 4 minutes. Added deduplication with task-ID idempotency keys.

Prompt Drift Across Backends

Same task type, different system prompts on VPS vs Mac due to stale config sync. Outputs diverged subtly. Centralized prompt registry on VPS, backends pull at task start.

Graceful Degradation Gap

When Backend B went offline for maintenance, queued tasks waited. But the dispatcher had no concept of "reroutable" tasks. Some Mac tasks could have run on VPS with reduced capability. Added fallback affinity as an optional field in the plugin registry.

What Made It Non-Trivial

"Route tasks to different machines" sounds like a solved problem. Load balancers exist. The hard parts are specific to LLM-based systems:

  • Tasks aren't homogeneous. A load balancer distributes identical requests. These tasks have fundamentally different runtime requirements — API keys, browser engines, GPU memory. Routing requires understanding what the task needs, not just which server is least busy.
  • LLM routing is a trap. It feels natural to use an LLM to classify tasks. But classification with known categories is a lookup problem, not an inference problem. Every token spent routing is a token not spent executing.
  • Auth between machines is mandatory. The moment you split work across backends, you need request signing. HMAC is the minimum viable approach — no session state, no token exchange, just a shared key and a timestamp.
  • Observability changes shape. Single-machine: check the logs. Multi-machine: which machine? Was the task dispatched? Did it arrive? Did the signature validate? Did it execute? Did the result make it back? Every hop is a failure point.

The 37-second cycle time is the result, not the goal. The goal was reliability across heterogeneous backends. Speed came from removing the LLM from the routing path and letting each backend do what it's good at.
