When More Agent Work Costs Less

API-based agents bill per token. Subscription plans don't. How we split orchestration from execution and cut costs by 10x overnight.

The cost problem

We were spending $376 on API calls across 26 days of building Briu. That sounds cheap until you project it forward. A production agent running daily workflows (email triage, CRM updates, report generation, content drafts) would burn through that in a week. Double the workload, double the bill. The API meter never stops.

This is the standard architecture for agent systems right now. You call the API, you pay per input token and per output token. Opus-class models run roughly $15 per million input tokens and $75 per million output. A single complex task (reading context, reasoning through steps, generating output) can easily hit 100K tokens. At scale, the math gets uncomfortable fast.

The more useful your agent becomes, the more it costs to run. That's the wrong incentive curve for a company selling agent-powered services.

The insight

Claude Max and Claude Pro are subscription plans. $200/month and $100/month respectively. Flat rate. Within the plan's usage limits, you can run as many conversations, as many tokens, as many tasks as you want. The marginal cost of the next task is zero.

Claude Code ships with both plans. It's a full terminal-based agent that can read files, write code, run commands, and iterate on its own work. It uses the same Opus and Sonnet models as the API, but billed against the subscription instead of per-token.

The gap between API pricing and subscription pricing creates a straightforward arbitrage. Any work you can move from API calls to subscription-based execution costs less. Move enough of it, and total cost drops even as total work increases.

The question was whether we could build a system that captures that gap reliably.

The architecture

Two machines, one protocol. The VPS orchestrates. The Mac executes.

The VPS runs the scheduling layer. It knows what needs to happen and when. When a task is ready, it builds a manifest: a JSON document describing the work, the target files, the expected outputs, and the constraints. Every manifest gets an HMAC-SHA256 signature using a shared secret. No valid signature, no execution.

The Mac receives manifests over SSH, validates the signature, and hands the work to Claude Code. Claude Code does the actual reasoning, file manipulation, code generation, and iteration. All of that runs against the Max subscription. Zero marginal cost.

Manifest structure

Each manifest is a signed JSON payload. The structure is deliberately simple so there's minimal surface area for things to go wrong.

  1. Task ID and timestamp. Every manifest gets a unique ID and a UTC timestamp. The executor rejects anything older than 5 minutes to prevent replay attacks.
  2. Plugin name. Maps to a specific skill in the agent's repertoire. "email-triage", "crm-sync", "content-draft", "deploy-check". The executor only runs plugins it knows about. Unknown plugin names get rejected.
  3. Parameters. Task-specific inputs. For email triage: the inbox to scan and the rules to apply. For content drafts: the topic, tone, and target length. Parameters are validated against the plugin's schema before execution.
  4. HMAC-SHA256 signature. Computed over the canonical JSON (sorted keys, no whitespace). The shared secret never leaves the two machines. If a single byte of the manifest changes, the signature fails.
  5. Execution constraints. Max runtime, allowed file paths, network access flags. The executor enforces these as hard limits regardless of what the model tries to do.

The dispatch is mechanical. No AI in the routing layer. The VPS doesn't interpret tasks or make judgment calls about what to run. It follows a schedule, builds a manifest, signs it, sends it. The intelligence lives entirely on the execution side, running on a flat-rate plan.

Same-day red team

We shipped the architecture in the morning and red-teamed it by afternoon. Same approach as our sanitizer red team: Opus attacking its own system, trying to find ways through.

The attack surface is narrow by design. The only entry point is the manifest, and the only thing the executor trusts is the HMAC signature. But narrow doesn't mean safe. We tested:

Manifest tampering. Modified payloads with valid-looking structures but invalid signatures. All rejected. The HMAC check runs before any parsing of the manifest body, so malformed JSON with a bad signature never even gets deserialized.

Replay attacks. Captured a valid signed manifest and resent it after the 5-minute window. Rejected on timestamp. Resent it within the window. The executor's idempotency check caught the duplicate task ID.

Plugin injection. Crafted manifests with plugin names that don't exist in the allowlist. Rejected. Tried plugin names with path traversal characters (../../etc/passwd as a plugin name). Rejected at schema validation before the name ever hits the filesystem.

Parameter overflow. Sent manifests with parameters designed to make the model behave unexpectedly. Extremely long strings, nested objects 50 levels deep, Unicode edge cases. The schema validator catches structural issues. For content-level attacks (prompt injection via parameters), the execution constraints prevent the model from taking actions outside its allowed file paths and network scope.

The result: zero bypasses on the dispatch layer. The architecture's security comes from its simplicity. There's no AI making trust decisions in the routing path. It's HMAC or nothing.

What this means for clients

The cost arbitrage flows directly to the service. When we deploy agent workflows for a client, the execution cost doesn't scale linearly with usage. Adding a second workflow to an existing deployment doesn't double the cost. Adding a third barely registers. The subscription absorbs the work.

This is why Briu can offer agent-powered services at a price point that doesn't make sense if you assume API-rate economics. We're not eating the cost. The cost structure is genuinely different.

More work. Less cost. That's the point of the whole company.
