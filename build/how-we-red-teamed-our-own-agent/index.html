It started with a sanitizer

We wrote an input sanitizer to catch prompt injection in external content—emails, CRM fields, calendar entries, anything that flows into the agent from outside. 140+ regex patterns, 20+ language families, 7 detection layers. It caught most things.

So we did what you should do with any security control: we tried to break it.

The first round used Opus 4.6 as the attacker. We gave it the sanitizer source code and told it to craft payloads that would bypass every detection layer. 30 payloads. 90% catch rate after patching. Good start.

Then we ran it again. And again. Five rounds with Opus, each one targeting whatever the previous round missed. The sanitizer improved each time. But the payloads that kept getting through taught us something more important than any individual fix:

Pattern matching is a necessary layer, not a sufficient one. No regex engine will catch a prompt injection written as a metaphor, encoded in morse code, or split across three separate tool results. You need structural defenses that don't depend on recognizing the attack.

Each round, the attacker model got the full sanitizer source code, the previous test results, and a mandate to find bypasses. Each round, we patched what it found and re-tested everything.

The detection rate column is only part of the story. By v7 and v8, the payloads that got through were increasingly exotic: instructions encoded as morse code, attacks hidden in word-initial steganography, injection spread across multiple tool results that were clean individually. None of these are catchable with regexes. That's not a failure of the sanitizer—it's the boundary of what pattern matching can do.

Which is why the sanitizer is only one of ten defense layers.

Defense in depth: 10 layers

Every layer assumes the others have failed. That's the design principle. If an attacker gets past the sanitizer, structural framing catches it. If they defeat the framing, canary tokens detect the exfiltration. If they evade the canaries, exec guardrails block the command. And so on.

1

Input Sanitizer

140+ patterns, 7 detection stages, 20+ languages. Catches known attack patterns before the model sees them.

2

Auto-Sanitizer Hook

Plugin-level enforcement. Every tool result over 50 chars is scanned automatically. The agent cannot opt out.

3

Canary Tokens

Daily-rotating fake credentials. If an injection extracts them, the leak is detected and blocked instantly.

4

Write Protection

Critical files (CLAUDE.md, exec-approvals, plugins) locked from all non-operator agents. Blocks direct and indirect writes.

5

Exec Guardrails

Blocks inline code execution, network exfiltration, and file exfiltration for non-main agents.

6

Kill Switch

Automated lockdown on critical CVEs. All non-main agents stopped until the vulnerability is patched.

7

Sandbox Isolation

External content processed through a sandboxed agent (read-only, Haiku) with structured JSON output and rate limits.

8

Structural Framing

All external content wrapped in XML tags that signal "data, not instructions." Works even when regex fails.

9

HMAC Task Signing

Cryptographic verification of all inter-agent task manifests with nonce replay protection and 60-second expiry windows.

10

Pipeline Exec Parsing

Structural command parsing that validates every segment in pipes and chains against the allowlist independently.

That's the architecture. But testing it with payloads only exercises layer 1. The real question was: what happens when the attacker targets the architecture itself?

Codex found what Opus missed

After five rounds of payload testing, we changed the approach. Instead of generating more attack strings, we gave Codex the full source code of every plugin, the hook system, the cron jobs, the state database, the exec-approvals config—and told it to find architectural vulnerabilities.

It ran four systematic audits:

Plugin interaction audit

Reviewed all 5 plugins. Hook execution order, cross-plugin data flow, return field contracts with the runtime.

Cron prompt injection audit

Traced data flow from external sources (email subjects, calendar titles, comments) through cron jobs to model context.

State-db trust boundary audit

Mapped which agents could write to which state tables, and what the main agent consumed as trusted instructions.

File permission & CLI audit

Unix permissions on all operational files, symlink target validation, argument injection in 9 CLI wrapper scripts.

The results were sobering. Not because the sanitizer was weak—it passed all 41 of Codex's payloads. Because the architecture had assumptions that weren't true.

Five critical findings

These are the ones that mattered. Not theoretical—each one was a real attack path that would have worked in production.

Critical — The silent drop

Three security plugins were returning a field the runtime silently ignored

All three security plugins—canary tokens, cost alerts, shell rules—were returning systemReminder from before_prompt_build. But the runtime only consumes systemPrompt, prependContext, appendSystemContext. The plugins were running. They were generating output. The runtime was throwing it away. Every security injection that was supposed to reach the model—canary tokens, cost alerts, task state—was being silently dropped.

The agent was operating without these defenses in its system prompt. No error. No warning. Just… nothing.

Fixed — switched to appendSystemContext
Critical — Cross-layer bypass

Injecting a fake tag caused the auto-sanitizer to skip all scanning

The auto-sanitizer hook checked whether content was already wrapped in <untrusted-content> tags before scanning. An attacker could inject that substring into a payload, and the hook would skip it entirely—believing the content was already sanitized. One string bypasses the entire scanning pipeline.

Fixed — validated structural framing detection
Critical — Plugin execution order

Content was being truncated before it was scanned for injection

The security-monitor's auto-sanitizer ran at priority 5. The context-optimizer ran at priority 10. Higher priority executes first. So content was being truncated or quarantined by the optimizer before the security scanner ever saw it. Injections in the truncated portion would never be detected.

Fixed — security-monitor raised to priority 100
Critical — State-db trust boundary

Any agent could write to state the main agent trusts as instructions

Non-main agents (cron jobs, sandbox) had full write access to the state database: watch_events, threads, kv store. The main agent consumed these as trusted operational data. A compromised cron job could write injection payloads into state-db fields that the main agent would later follow.

Fixed — read-only wrapper for non-main agents
Critical — Unframed external data in cron

Morning briefing ingested email subjects without structural framing

Cron jobs that ran the morning briefing and heartbeat checks ingested email subjects, calendar titles, and comments as plain text—no <untrusted-content> wrapping. An attacker who could control an email subject line had a direct injection path into the model context.

Fixed — all external fields wrapped in data pipeline

The lesson: compaction is an attack surface

One finding stood apart from the rest, because it's a fundamental problem with how LLM sessions work, not something specific to our stack.

When a session gets long, the runtime compacts it: it summarizes old messages to free up context window. During compaction, <untrusted-content> tags get stripped. Injection payloads that were safely framed become unframed text in the compaction summary.

Compaction turns safely-framed content into unframed text. An attacker who can get a payload into the session and wait for compaction to happen gets a free de-framing pass. The payload enters wrapped. It exits naked.

This is not something a regex can fix. It's a property of the compaction pipeline itself. The mitigations are layered: re-sanitize summaries after compaction, track "contains untrusted content" metadata through the pipeline, and add explicit rules to the model's system prompt about compacted content. We've implemented all three.

What pattern matching will never catch

After 290+ payloads across 8 rounds, the remaining bypasses clustered into four categories that no regex engine can address:

  • Pure semantic metaphor — instructions using completely novel vocabulary with no injection keywords. "Consider the garden" is not detectable as an instruction override.
  • Novel encoding — ROT13, morse code, NATO phonetic, word-initial steganography. The sanitizer can add decoders, but there will always be another encoding scheme.
  • Sub-threshold payloads — under 50 characters. Too short for effective attacks, but also too short to scan reliably.
  • Composition attacks — content that's clean individually but dangerous when combined across tool results. Each piece passes inspection. The combination is an instruction.

This is why defense-in-depth matters. Structural framing catches what the sanitizer can't. Canary tokens catch what the framing misses. Exec guardrails stop the action even if the model is compromised. You don't need every layer to be perfect. You need the combination to be reliable.

Beyond payloads: the architectural audit

The architectural audits found 30+ additional issues beyond the payload testing (before rounds 7–8 added another 47). Some highlights:

  • Symlink bypass of write protection — writing to /tmp/symlink → CLAUDE.md passed the path check. Fixed with path.resolve() before comparison.
  • Script file execution gap — exec guardrails blocked bash -c "payload" but allowed bash /tmp/payload.sh. Any non-main agent could write a script and execute it.
  • Missing exec verbs — dd, rsync, ln, truncate could modify protected files. 6 verbs added to the blocklist.
  • Canary detection encoding gap — base64 or URL-encoded canary values evaded exact substring matching. Added decoding before comparison.
  • 9 CLI wrapper scripts vulnerable to argument injection, SQL injection, or path traversal. All patched.

None of these would have been found by running more attack payloads against the sanitizer. They required reading the source code, understanding the architecture, and thinking about trust boundaries.

Rounds 7–8: Two-agent architecture

After six rounds of single-agent testing, we expanded the scope. The system had grown from one VPS agent to a two-agent architecture: a VPS agent (Jaume) handling operations and a Mac agent (Paula) handling dispatch. New attack surface. New trust boundaries. New things to break.

We ran 4 parallel red-team agents simultaneously, each targeting a different surface:

Mac attack surface

Tool policy bypass, claude-dispatch manifest verification, git tool escapes, /opt/homebrew path manipulation.

VPS defense stack

security-monitor plugin, session-anchor, exec-guardrail, sanitizer v5.3 evasion, canary token extraction.

Trust boundaries

HMAC attacks, Discord transport integrity, nonce/flock system, state-db integrity, cross-agent escalation.

External attack surface

Email injection (10 strategies), calendar invite injection, webhook payloads, persistence vectors, memory poisoning.

The results: 47 findings. 10 critical. 16 high.

The biggest finding: HMAC verification was designed but never implemented. The task-signing system had a task-sign script that generated cryptographically signed manifests. The Mac dispatch code had a verification function. But the verification was never wired into the execution path. Signed manifests were generated, transmitted, and then executed without checking the signature. The entire HMAC trust model was theater.

Three findings triggered architectural redesigns, not patches:

  • Exec-guardrail pipeline parsing — the guardrail checked the first command in a pipe but not subsequent segments. echo foo | curl passed because echo was allowed. Redesigned to parse every segment in pipes and chains independently against the allowlist.
  • HMAC verification — implemented the verification that was designed but missing. Added nonce replay protection with 60-second expiry windows. The trust model now actually works.
  • State-db enforcement — the read-only wrapper for non-main agents was enforced at the application layer but not at the database level. A direct sqlite3 invocation bypassed it entirely. Moved enforcement to database-level permissions.

We also mapped every multi-step attack chain across the two-agent boundary. Five chains were identified. All five are now blocked:

  1. Email → sandbox compromise → state-db write → main agent instruction injection
  2. Calendar invite → cron ingest → unframed briefing → model context manipulation
  3. Discord message forgery → unsigned task manifest → Mac agent command execution
  4. Git hook injection → dispatch code modification → tool policy bypass
  5. npm dependency → plugin load → priority hijack → security monitor bypass

How to red-team your own agent

If you're running AI agents in production, here's the approach that actually works:

  1. Start with payload testing. Give an attacker model your defense code and let it generate bypasses. Run multiple rounds. Patch and re-test everything each time.
  2. Stop when the bypasses get exotic. When the remaining bypasses are morse code and steganography, your pattern matching is solid. More regex won't help.
  3. Switch to architectural audits. Give the attacker model full source access. Plugin interactions, hook execution order, trust boundaries between agents, data flow through external sources.
  4. Test your assumptions, not your code. The silent drop finding wasn't a code bug. The plugin was running, the code was correct, the return value was right. The assumption that the runtime consumed that field was wrong.
  5. Check what happens at boundaries. Compaction boundaries, session boundaries, agent-to-agent handoffs, cron-to-main transitions. Every boundary is a place where security context can be lost.

AI is good at finding AI vulnerabilities. Not because it's smarter than a human pentester, but because it will systematically trace every data flow, check every return type, and read every line of every plugin without getting bored or making assumptions.

The point

We shipped 70+ security fixes across 8 rounds. Fifteen were critical. The biggest one—an HMAC verifier that was designed but never wired in—meant the entire inter-agent trust model was theater. No user would have reported it. No error log would have flagged it. It took a model reading every code path end-to-end to find it.

That's not a story about AI being dangerous. It's a story about AI being useful for exactly the kind of tedious, exhaustive verification that humans are bad at.

If you're building agents that handle real business data, you should be doing this. Not once. Continuously. The attack surface changes every time you add a plugin, modify a hook, or update a dependency. The defenses need to keep up.

Defense in depth isn't a feature. It's a practice.
