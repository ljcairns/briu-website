The measurement bug

Halfway through the session, our cost watchdog reported $107.01. We nearly started optimizing around that number. Instead, we audited the tool.

The bug: OpenClaw's compaction process creates a derived session file containing all previous messages. Our cost aggregation was scanning all JSONL files and summing by date without deduplicating by message ID. After compaction, every message got counted twice.

The fix was a few lines: deduplicate by message id across session files, process primary sessions before derived ones.

Lesson: A cost tracker that double-counts is worse than no cost tracker. When the watchdog reported $107, the instinct was to start reasoning about why the session was so expensive. The correct response was to audit the measurement tool first. Any aggregation over mutable, derived, or compacted data needs deduplication logic, or the numbers are untrustworthy by construction.

Understanding the cost structure

Cache operations are 89% of the total spend. The output, the actual model reasoning and writing, is 10%. This is the fundamental insight about AI session economics: you are paying for memory, not thinking.

The Anthropic pricing for Claude Sonnet 4.6:

  • Cache write: $3.75 per million tokens
  • Cache read: $0.30 per million tokens
  • Output: $15.00 per million tokens

At 180,000 tokens of context, every turn that causes a cache write costs approximately $0.68 just to touch the cache before a single output token is billed. At 20,000 tokens, that same turn costs $0.08. The 8x difference comes entirely from context size.

The 05:00–06:00 PT hour was the most expensive single hour. Reading 14 large source files in sequence, each invalidating the cache, each re-warming at 180K tokens. The hour totaled ~$19, but the cause was not "source analysis is expensive." The cause was: the session had already grown to 180K tokens before the source analysis started.

The correct pattern

Any large-file analysis should happen in a fresh subagent context:

sessions_spawn(
    agentId="haiku",
    task="Read /path/to/large/file and return: [specific question]",
    runTimeoutSeconds=60
)

A full source analysis that costs $4.00 in the main session at 180K costs approximately $0.20 via subagent delegation. The savings are from context isolation, not from the model being cheaper per token.

The cost optimization framework

The session's cost postmortem had an obvious next step: if uncontrolled context growth is the cost multiplier, build controls. The result was the Context Logistics Layer.

The core contract: the main session coordinates. It does not do leaf work. Any task requiring more than 2-3 tool calls should be scoped, budgeted, and delegated to a subagent.

Budget class system:

ClassToken BudgetUse Case
Quick5KSingle-tool, single-file, trivial operations
Bounded10-15KFile edits, targeted code changes
Research20-25KWeb search, multi-source reading
Analysis25-30KSource inspection, debugging, audit
Orchestration10-15KSubagent coordination, handoff

Mandatory subagent routing — any of these task types goes to a subagent, no exceptions:

  • 3 or more source files to inspect
  • Minified or compiled bundle inspection
  • JSONL or log analysis
  • QA audits
  • Multi-source research

The architectural takeaway:

Main session = coordination
Leaf work = subagents
Context does not accumulate across work items

Operational lessons

1. Context size is the cost multiplier, not individual operation cost. The 8x difference between a 20K and 180K session is entirely from context growth. Keep sessions short and compact aggressively.

2. Compact earlier than you think. A misconfigured softThresholdTokens let this session run to 180K when it should have flushed at 80K. Hours of expensive turns that could have been cheap.

3. Transcript mutation is not delivery mutation. If you build a plugin that modifies message content, you need to know which pipeline your channel uses. Discord delivery and JSONL storage operate on different objects.

4. Read static files once. Re-reading a large reference file invalidates the cache and costs a full re-warm. Once it is in context, reference it from memory.

5. Batch gateway restarts. Each restart costs one full cache re-warm on the next turn. Write all changes first, then restart once.

6. The model should not author operational metadata. Cost figures, deployment status, API responses should come from the system. The placeholder contract exists so the model cannot hallucinate a cost figure.

7. When you have four cost surfaces, you have zero cost visibility. The briefing, the watchdog, the dashboard, and the interactive block should all read the same canonical state.

8. Audit the measurement tool before trusting the number. When the watchdog reported $107, the instinct was to optimize. The correct response was to verify the tool first.

The thesis

Building a system you can trust is not primarily a question of capability. It is a question of evidentiary discipline.

Gate's result schema is not bureaucratic overhead. It is the difference between a system that tells you it worked and a system that shows you it worked.

The cost block placeholder is not a design nicety. It is a commitment that the system will not invent operational facts.

The security three-bucket model on the website is not aesthetic. It is the same principle applied to public claims: say what you measured, not what you imply.

Everything we built in this session is about building a system that does not require you to trust it blindly. It provides evidence. You verify the evidence. Trust is a conclusion, not a premise.

That is the actual product we are building.
