What happened

This is not a post about adding cost alerts to an AI system.

It is a post about what happened when we tried to do that and discovered the system had three separate cost pipelines, a runtime hook assumption that was silently wrong, a delivery architecture where transcript mutation and outbound message delivery operate on entirely different objects, and a cost tracker that had been double-counting its own session data.

It is also a post about a session that initially appeared to cost $107. After fixing a double-count bug in the measurement pipeline, the actual operational cost was $64.30.

The session started somewhere else

We did not sit down at 1 AM to work on cost monitoring.

We started with a review of briu.ai, the public-facing site for this project. That turned into something more disciplined than a visual pass.

The problem with the site was not aesthetic. It was evidentiary. The Build page was making claims that were either too broad or not sourced from anything real:

  • The commit counter was hardcoded at 625. The actual number was 641 and climbing.
  • The skills count said "12 skills." The live config had 9.
  • The cost figure said "$2–5/day." The actual recent daily average was $14.68.
  • "Security" was presented as a vague implied property, not as a set of tested and verifiable claims.

This is the kind of site content that accumulates naturally when you're moving fast. But for a site that's supposed to demonstrate that we build trustworthy systems, it was the wrong kind of wrong.

So we fixed the data layer first.

The stats centralization pass

dynamic-stats.js got a FALLBACKS object, a single source of truth for all default stat values. The org chart now pulls commit counts from the live stats.json file rather than hardcoded strings. Agent skills link to the live site-config.json. The daily cost figure is labeled as an average with a source.

The commit label distinction matters enough to spell out:

  • Project commits: total across website and agent configuration repos
  • Website commits: briu-website repo only
  • Agent commits: private configuration repo

These are not interchangeable. A site that conflates them is doing the same thing as an AI that treats all cost data as equivalent regardless of source.

The security metrics redesign

The old security section implied broad coverage without committing to specifics. After two passes, it became a three-bucket model:

Architecture — what structural properties the system has: no persistent memory across untrusted boundaries, role separation, scope-bounded approvals.

Adversarial testing — what we have actually tested: 26 known injection payloads, 15 attack categories, zero bypasses found in the current surface.

Operating posture — what ongoing controls are in place: daily cost ceiling, monthly audit cycle, prompt canary tokens.

The restructure came from a single push: stop claiming properties you haven't measured. If you have tested 26 payloads, say 26, not "robust." If you haven't tested something, don't imply coverage.

This is not just website discipline. It is how we want the agent to reason about its own security properties.

Then we built the Gate

Somewhere around 3 AM, we moved to the execution safety layer.

The Gate is a separate named agent in the OpenClaw configuration. It is not a skill. It is not a plugin. It is a persistent subprocess with its own system prompt, its own exec-approvals allowlist, and one specific role: receive a JSON plan from the main agent (Claw), validate it against a scope declaration, execute the approved commands in order, and report structured results.

The design principle is clean and worth saying plainly:

Claw reasons. Gate executes. They are not the same agent.

Why this matters architecturally

When a single agent both plans an action and executes it, there is no boundary. It is easy for the planning logic to quietly expand what "this command" means at execution time. It is easy to declare success without verification. It is easy for a model to say "I ran the command" when it simulated running the command.

Gate's job is to enforce an evidence boundary. Every result it returns has a required schema:

STATUS:    success | partial | failed | unverified
INTENDED:  what Claw asked for
EXECUTED:  what actually ran
VERIFIED:  what the output confirmed
EVIDENCE:  literal output proving the claim
OPEN RISK: anything unverified or uncertain
ARTIFACT:  any file written or state changed

The schema exists because "it worked" is not an engineering claim. "I ran git log -3 and the top commit is 8be3856 with message security three-bucket model" is an engineering claim.

The tier model

Every command in a Gate plan has a tier:

  • Tier 0 — read-only, no side effects, auto-execute. git status, git diff, file reads, process inspection.
  • Tier 1 — local mutation, reversible, requires scope grant once per session. File edits, git commit within approved branch.
  • Tier 2 — boundary-crossing, always requires fresh approval. Push, deploy, external service call.
  • Tier 3 — never. No command list. Gate refuses.

The scope declaration travels with the plan. It specifies the repository, the branch, and what mutations are allowed. plan_hash binds the approval to the exact plan, any modification after approval invalidates it. Gate never infers the push branch at runtime.

The Tier 0 test

After the Gate was registered and its exec-approvals allowlist was loaded, we tested it.

We spawned Gate as a subagent via sessions_spawn(agentId="gate") and sent it a plan with two Tier 0 steps: git status and git log -3.

Both ran without any approval prompts. Gate returned exit 0, structured output, and the evidence hash. The test passed.

This is not glamorous. But it is the foundation everything else sits on. The first time a bounded agent executes correctly in a controlled environment is the proof-of-concept moment for the whole tier model.

Then the cost monitoring arc started

This is where the session turned expensive and interesting.

The immediate trigger was the cost watchdog. We had a Python script that ran every 15 minutes, read the session JSONL, computed the running daily cost, and posted to Discord when thresholds were crossed. It had four bugs:

  1. State lived in `/tmp/` — not persistent across reboots. Every restart reset the alert history.
  2. Threshold batching was broken — if the session crossed $15 and $20 thresholds in the same 15-minute window, only the last one was reported.
  3. The usage_report API bucket_width was wrong — the fallback from 1h to 1d was not implemented, so the API would return no data during a certain window.
  4. No diagnostic logging — when the watchdog failed silently, there was no way to know why.

All four were fixed. State moved to ~/.openclaw/cost-watchdog-state.json. Threshold batching now posts all crossed thresholds in ascending order in one message. The bucket_width=1h fallback is in. The logs write to /tmp/openclaw/cost_watchdog.log.

These were clean, contained fixes. The harder problem was the question they surfaced:

Where does the cost number actually come from?

Three cost pipelines, none unified

By mid-session, we had mapped the cost data landscape:

Consumer Source Boundary Live?
Watchdog alerts Session JSONL PT calendar day Yes, but JSONL-based
Morning briefing Anthropic Billing API UTC calendar day Yes, accurate
Dashboard SPEND strip openclaw gateway usage-cost JSONL PT calendar day Yes, estimate
Interactive cost blocks Anthropic usage_report API UTC rolling 1h/1d Not yet working

The briefing showed "$21.32 yesterday." The dashboard showed "$6.51 yesterday" for the same day. The difference was not a bug in either system. It was a combination of different sources and different timezone boundaries, billing API uses UTC days, the JSONL-based paths use local PT dates.

This is the kind of confusion that compounds. A system monitoring its own costs needs one number, one source, and one clearly labeled freshness state. Having four separate computations is how you end up debugging a cost alert instead of trusting it.

The response was cost_canonical.py — a unified collection module that:

  • Polls openclaw gateway usage-cost --days 2 --json (JSONL-based, real-time) as the primary source
  • Falls back to the billing API snapshot for delta tracking
  • Writes a canonical state file at ~/.openclaw/cost-canonical-state.json
  • Exposes --format inline|full|tier|state
  • Returns a freshness label: fresh (≤20 min), stale (>20 min or today absent), or unavailable

The watchdog now writes through to this canonical state file after each cycle. All consumers will eventually read from the same file.

The usage_report API has an important property we had to document explicitly: it has approximately a 24-hour lag. Today's data does not appear in today's API response. The JSONL-based estimate from gateway usage-cost is real-time but approximate (estimated from token counts at fixed rates). The billing API is accurate but yesterday-only.

This means "how much have I spent today" is inherently an estimate until tomorrow. The freshness label exists precisely to communicate that.

The placeholder design and the assumption that broke it

Once the canonical pipeline existed, the question became: how does the model communicate cost information in a response without fabricating numbers?

The answer was a placeholder contract:

[[COST_BLOCK:inline]]   → single-line summary
[[COST_BLOCK:full]]     → multi-line block with token breakdown

The model is instructed never to author cost figures directly. It writes the placeholder. The delivery layer replaces it with the rendered block from the API.

This is a clean design. The model cannot hallucinate a cost figure because the model never writes cost figures.

The implementation question was: where does the replacement happen?

v5: message_sending hook, failed

The first attempt used the message_sending plugin hook. In theory, this fires just before a message is sent to the user, and the handler can modify the content. The model writes [[COST_BLOCK:inline]], the hook replaces it with 💰 $47.20 today [fresh], and the user sees the rendered block.

It did not work on Discord.

The investigation took a while. The hook exists and is correctly documented. It fires reliably for Telegram and the generic outbound path. But Discord uses a separate delivery path:

runEmbeddedPiAgent
  └── createReplyDispatcherWithTyping
        └── deliver: async (payload, info) => {         ← no hook here
              └── deliverDiscordReply({replies:[payload]})
                    └── chunkDiscordTextWithMode(payload.text)
                    └── sendDiscordChunkWithFallback({text: chunk})

message_sending lives inside deliverReplies(), which is the Telegram path. The Discord delivery function deliverDiscordReply is called directly from the deliver callback in the streaming dispatcher. deliverOutboundPayloads(), which invokes message_sending, is never called.

We confirmed this by reading approximately 14 source files from the OpenClaw compiled bundle. That source inspection, done inside the main session at 180,000 tokens of context, is the proximate cause of the session's largest cost spike.

v6: before_message_write hook, partially correct

The second attempt used before_message_write, a synchronous hook that fires just before an assistant message is appended to the session JSONL transcript.

This hook fires correctly. It modifies the message object that gets stored. The JSONL has the rendered block.

But Discord still showed the literal placeholder text.

The reason is subtle and important:

Transcript mutation and delivery mutation operate on different objects.

The streaming dispatcher assembles payload.text during the LLM run. By the time before_message_write fires (at JSONL append time), the Discord delivery has already consumed its own copy of the payload assembled earlier in the pipeline. These are independent objects with no back-reference.

LLM streaming → payload.text assembled (snapshot A)
                        │
                        ├─── Discord delivery: deliverDiscordReply(payload)  ← uses snapshot A
                        │
                        └─── Session storage: before_message_write fires → modifies JSONL entry
                                                                             ← uses snapshot B (separate object)

The model writes [[COST_BLOCK:inline]]. Discord receives [[COST_BLOCK:inline]] (unmodified snapshot A). The JSONL stores the rendered block (modified snapshot B). The two are out of sync.

This is v1 behavior: deterministic rendering in the transcript, literal placeholder in Discord.

The correct fix requires a hook at the deliver callback layer, inside createReplyDispatcherWithTyping, before deliverDiscordReply is called. This hook does not exist in the current runtime. We have named it before_reply_deliver and documented the five-line gateway patch that would add it.

Until that patch lands, the v7 interim approach is: pre-render in before_prompt_build (which fires before the LLM run) and instruct the model to copy the rendered block verbatim. Both Discord and JSONL receive the model's output, which includes the pre-rendered text. This works, but it is model-compliance-dependent, not a delivery hook. It is labeled clearly as an interim implementation.

The session cost postmortem

At some point around 10 AM, we turned the session cost into the subject of its own analysis. It became one of the more instructive parts of the day, not because the spending was surprising, but because measuring it correctly turned out to be harder than expected.

The double-count bug

The cost watchdog initially reported $107.01 for the session. This was wrong, and the bug that caused it is worth understanding in detail.

OpenClaw's compaction process, which flushes long sessions to a summary to control context size, creates a new derived session file. That file contains all previous messages from the original session up to the compaction point, with a parentSession pointer back to the original.

The cost aggregation script was scanning all JSONL session files and summing costs by date. It was not deduplicating by message ID. When compaction ran at 11:38 AM, a derived session file appeared containing every March 10 message that had already been counted in the primary session. Every message was summed twice.

The compaction snapshot contained the entire message history of the original session. The cost pipeline counted both the original session and the snapshot, doubling the spend.

The fix was straightforward: deduplicate by message id field across all session files, processing primary sessions (UUID-named filenames) before derived sessions (ISO-timestamp-named filenames) so primary IDs are registered first and derived duplicates are discarded.

Corrected figures:

  • $64.30 operational today (PT) — JSONL deduplicated, grouped by PT date
  • $70.98 billing estimate today (UTC) — same data, grouped by UTC date to match Anthropic's billing boundary
  • $107.01 — the inflated double-counted figure reported before the fix

The $42.71 gap was pure duplication noise. The real session cost is $64-70 depending on which day boundary you use.

The actual cost driver: context size, not individual operations

The corrected breakdown by cache type:

Bucket Cost %
Cache write (5m+1h TTL) $34.36 53%
Cache read $23.37 36%
Output $6.47 10%
Input $0.10 0%

Cache operations are 89% of the total spend. The output, the actual model reasoning and writing, is 10%.

The Anthropic pricing for Claude Sonnet 4.6:

  • Cache write: $3.75 per million tokens
  • Cache read: $0.30 per million tokens
  • Output: $15.00 per million tokens

These rates are the same for 5-minute and 1-hour TTL cache writes. The portal shows them in separate audit rows, but the per-token cost is identical.

The important number: at 180,000 tokens of context, every turn that causes a cache write costs approximately $0.68 just to touch the cache before a single output token is billed.

This is not a per-operation cost. It is a per-turn floor that scales with context size.

  • A session at 20K context: floor per turn, approximately $0.08.
  • A session at 180K context: floor per turn, approximately $0.68.

The cost multiplier from context growth is roughly 8x. Not from doing more expensive things, from doing the same things in a larger context window.

The 05:00-06:00 PT hour was the most expensive single hour. Reading 14 large source files in sequence, each one invalidating the cache, each invalidation re-warming at 180K tokens: $0.68 times 14, approximately $9.52 in cache writes before a line of analysis output. The hour totaled roughly $19, but the cause was not "source analysis is expensive." The cause was: the session had already grown to 180K tokens before the source analysis started. The re-warm cost was expensive because the context was large, not because the files were large.

This is the actual lesson.

The correct pattern

Any large-file analysis should happen in a fresh subagent context:

sessions_spawn(
    agentId="haiku",
    task="Read /path/to/large/file and return: [specific question]",
    runTimeoutSeconds=60
)

Haiku at 20K context: cache floor per turn, approximately $0.08. A full source analysis that costs $4.00 in the main session at 180K costs approximately $0.20 via subagent delegation. The savings are from context isolation, not from Haiku being cheaper per token.

The compaction threshold correction

The session also surfaced an error in how we'd configured the memory flush compaction threshold.

The formula is: flush_threshold = context_window - reserve_floor - softThresholdTokens

The flush fires when session_tokens >= flush_threshold.

We had changed softThresholdTokens from 60,000 to 40,000 believing this would make compaction more aggressive. It did the opposite: it raised the flush threshold from 100,000 to 120,000 tokens, meaning sessions would run longer before compaction.

The correct value for an 80,000-token flush threshold is softThresholdTokens = 80,000. This was corrected and hot-reloaded by the gateway config watcher (confirmed in gateway logs at 11:29 AM, no restart required for this field).

The response: a context logistics layer

The session's cost postmortem had an obvious next step: if uncontrolled context growth is the cost multiplier, build controls.

The Context Logistics Layer is a set of behavioral contracts, not a plugin, not a tool, that govern how the main agent manages session scope. It lives in AGENTS.md and TOOLS.md, which are loaded at session start.

The core contract: the main session coordinates. It does not do leaf work. Any task requiring more than 2-3 tool calls should be scoped, budgeted, and delegated to a subagent before starting.

The budget class system (in TOOLS.md):

Class Token budget Use case
Quick 5K Single-tool, single-file, trivial operations
Bounded 10-15K File edits, targeted code changes
Research 20-25K Web search, multi-source reading
Analysis 25-30K Source inspection, debugging, audit
Orchestration 10-15K Subagent coordination, handoff

Any work item must declare its budget class before starting. The main session stops new subtasks at 80% of the declared budget and writes a handoff.

Mandatory subagent routing — any of these task types goes to a subagent, no exceptions:

  • 3 or more source files to inspect
  • Minified or compiled bundle inspection
  • JSONL or log analysis
  • QA audits
  • Multi-source research

The source analysis that cost $19 in this session would have been budget_class: "analysis", spawned as a haiku subagent with a single targeted question. Main session stays at 20K. The analysis session runs in its own context and discards.

The handoff artifact: at every session boundary, the agent writes workspace/state/handoff.json, a structured continuation artifact with the active work item, in-progress state, pending tasks, and a 500-character carry field for critical context. A human-readable companion (SESSION-HANDOFF.md) ensures the state survives even if the JSON schema changes.

The schema's discard boolean is worth noting: when a subagent completes a task, its context should not be inherited by the main session on resume. Garbage collection is explicit and deliberate, not implicit.

The architectural takeaway in three lines:

Main session = coordination
Leaf work = subagents
Context does not accumulate across work items

This is the same principle as Gate applied to session economics. Gate prevents a single agent from both planning and executing without a boundary. The Context Logistics Layer prevents a single session from both coordinating and doing without a boundary. The pattern scales.

The mobile or Discord constraint as architectural input

A thread running through all of this is that the primary interface for this session was Discord on a phone.

This is not a minor detail. It shaped the architecture in several concrete ways:

Discord message fragmentation — responses longer than 2,000 characters get split. A long diagnostic output that makes sense as a single unit arrives as four separate messages. The workaround is smaller output blocks and better session discipline, not bigger messages.

Files trapped on the VPS — when Claw writes an artifact file, it exists at a VPS path. From a phone, that path is not accessible. The artifact rule we formalized: when creating a file Lucas may need to read or move from mobile, either attach directly in Discord, paste a Discord-safe version in blocks under 2,000 characters, or provide a short summary plus the VPS path. Never assume VPS-local files are accessible.

Copy or paste friction — verifying whether an output is correct from phone requires either trusting it or context-switching to a laptop. This is where "I want it to feel like a trustworthy employee" becomes a system requirement, not a product metaphor. Verification cannot require a terminal.

Thread boundaries as session boundaries — the realization that Discord threads should map to session boundaries, not just topic groupings. A new thread gives the model a clean context. An old thread is carrying accumulated cost weight from every prior turn.

These constraints were not solved in this session. They were understood better. Understanding them is the prerequisite for building the relay skill, the artifact delivery system, and the mobile handoff protocol.

The operational lessons, stated plainly

1. Context size is the cost multiplier, not individual operation cost. Every turn in a 180K-token session costs approximately $0.68 in cache writes before any output is billed. A session at 20K costs approximately $0.08 per turn. The 8x difference is entirely from context growth. The fix is not avoiding expensive operations, it is keeping sessions shorter and compacting aggressively. Subagent delegation for large-file analysis is valuable because it runs in a fresh 20K context, not because haiku is cheaper per token.

2. Compact earlier than you think. With a 100K-token flush threshold (original setting), a session that grew to 180K had 80K tokens of expensive context running for hours after compaction should have fired. At 80K threshold, the flush fires sooner and subsequent turns are cheaper.

3. Transcript mutation is not delivery mutation. If you are building a plugin that modifies message content, you need to know which pipeline your channel uses. before_message_write fires on the JSONL storage path. For Discord, delivery has already happened by then. These are different objects.

4. Read static files once. TOOLS.md, SOUL.md, MEMORY.md, plugin source, each of these was read multiple times during this session. Once a large reference file is in context, re-reading it invalidates the cache and costs a full re-warm. Use sed -n with exact line numbers for targeted updates.

5. Batch gateway restarts. Each restart between plugin edits costs one full cache re-warm on the next turn. Write all changes first, then restart once.

6. The model should not author operational metadata. Cost figures, deployment status, API responses, these should come from the system, not from the model's output. The placeholder contract exists for this reason. A model that authors a cost figure can be stale, hallucinated, or simply wrong in a way that looks like a real number. A system that injects a freshness-labeled API snapshot cannot be.

7. When you have four cost surfaces, you have zero cost visibility. The briefing, the watchdog, the dashboard, and the interactive block should all read the same canonical state. Until they do, you cannot trust any of them fully.

8. A cost tracker that double-counts is worse than no cost tracker. When the watchdog reported $107, the instinct was to start reasoning about why this session was so expensive. The correct response was to audit the measurement tool first. The double-count came from a basic data integrity failure: summing costs across JSONL files without deduplicating message IDs. The fix was a few lines. The lesson: any aggregation over mutable, derived, or compacted data needs deduplication logic, or the numbers are untrustworthy by construction.

The thesis, stated as an engineering claim

Building a system you can trust is not primarily a question of capability.

It is a question of evidentiary discipline.

Gate's result schema is not bureaucratic overhead. It is the difference between a system that tells you it worked and a system that shows you it worked.

The cost block placeholder is not a design nicety. It is a commitment that the system will not invent operational facts.

The security three-bucket model on the website is not aesthetic. It is the same principle applied to public claims: say what you measured, not what you imply.

Everything we built today, the Gate tier model, the cost visibility pipeline, the compaction threshold correction, the runtime hook investigation, is about building a system that does not require you to trust it blindly. It provides evidence. You verify the evidence. Trust is a conclusion, not a premise.

That is the actual product we are building.

Technical debt and remaining risks

Stated explicitly, not buried:

  • Discord cost injection is model-compliance-dependent until before_reply_deliver exists. The model copies the pre-rendered block from the system prompt. If it doesn't, the block is absent from Discord. This works reliably in practice but is not a delivery guarantee.
  • The Anthropic usage_report API has approximately 24-hour lag. "Today's cost" is always an estimate from JSONL until the billing API catches up the following UTC day.
  • Gate Tier 1 is not yet implemented. Tier 0 (read-only) is validated. File writes and commits require the scope-gated approval flow, which is designed but not yet built.
  • The crontab /tmp/new-crontab.txt install is pending. The watchdog runs on the old schedule until Lucas installs the updated crontab from a terminal.
  • The QA fixes to briu.ai are not committed. Changes are staged in /tmp/briu-redesign but not pushed. The site has not yet received the final messaging alignment pass.
