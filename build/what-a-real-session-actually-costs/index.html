It started as a site review

Most nights when that happens, I grab my phone and scroll through nothing for an hour. This time I opened Discord and started poking at the website. Not a planned session. Just, the site was on my mind and I had a phone and nothing better to do.

Within ten minutes I'd found four problems:

  • Commit counter hardcoded at 625. Actual number: 641.
  • Skills count said 12. Live config had 9.
  • Daily cost figure said "$2–5/day." Recent average was $14.68.
  • Security section implied broad coverage without committing to any specific claim.

None of these are catastrophic. They're the kind of thing that accumulates when you're moving fast. But for a site that's supposed to demonstrate trustworthy systems, stale numbers are the wrong kind of wrong.

So I told the agent to fix the data layer. Stats centralized through a single FALLBACKS object. Commit counts pulling from live stats.json. Security section rebuilt into three specific, measurable buckets: architecture properties, adversarial test results (26 payloads, 15 categories, 0 bypasses), and operating posture.

That took about an hour. I was still in bed.

Working from your phone at 2 AM

Here's the part that's hard to explain without experiencing it: this is genuinely engaging.

You send a message. You get a structured response with what changed and what it needs approval for. You read it, approve it or redirect it, and send the next thing. The back-and-forth has a rhythm to it. You're not babysitting. You're collaborating. And unlike doomscrolling, you're actually getting something done during hours you'd otherwise just be tossing and turning.

The trust model is what makes this work. The agent can read files, run diagnostics, draft changes. But anything that crosses a boundary—pushing code, sending an email, modifying a live service—requires explicit approval. From my phone, that's a Discord button. I see what it wants to do, I approve it, it does it, it shows me the result.

That approval flow is the entire product, really. Without it, you can't work from your phone because you can't trust what's happening. With it, you can lie in bed and ship real infrastructure.

The session pivoted organically

The site review surfaced stale cost numbers. That raised an obvious question: why don't we have a reliable cost pipeline?

Turns out we had four separate cost computations, three different timezone boundaries, and a watchdog script with four bugs. The briefing showed "$21.32 yesterday." The dashboard showed "$6.51 yesterday" for the same day. Neither was wrong—they were just reading different sources with different day boundaries.

So what started as "fix the numbers on the website" became "build a real cost monitoring system."

This is the thing about working with an agent from your phone. A casual review turns into a real implementation session, and the transition is seamless because you're already in the conversation. You don't need to context-switch to a laptop, open an IDE, set up a branch. You just keep talking.

Gate: the execution safety agent

Somewhere around 3 AM, we built Gate.

Gate is a separate agent. Not a skill, not a plugin. A persistent subprocess with its own system prompt and its own allowlist. It has one job: receive a plan from the main agent, validate it, execute the approved commands, and report structured results.

The principle is simple. The orchestrator reasons. The executor validates. They are not the same agent.

STATUS:   success | partial | failed | unverified
INTENDED: what was requested
EXECUTED: what actually ran
VERIFIED: what the output confirmed
EVIDENCE: literal output proving the claim

"It worked" is not an engineering claim. "I ran git log -3 and the top commit is 8be3856" is an engineering claim.

Commands are tiered. Tier 0 (read-only) auto-executes. Tier 1 (local writes) requires one scope grant per session. Tier 2 (pushes, deploys, external calls) always requires fresh approval. Tier 3 doesn't exist. Gate refuses.

The insight worth keeping

The most useful thing we learned from the cost analysis:

You're not paying for the model to think. You're paying for it to remember.

Cache operations (writing and reading the session context) were 89% of total spend. Actual reasoning and output generation: 10%.

At 180,000 tokens of context, every turn costs ~$0.68 in cache writes before any output. At 20,000 tokens, that same turn costs ~$0.08. Same operation, 8x cost difference, entirely from context size.

The most expensive hour ($19, from 5–6 AM) wasn't expensive because of what we were doing. It was expensive because the session had grown to 180K tokens before we started reading large files. Every file read re-warmed the full context.

The fix: delegate large-file analysis to subagents that run in their own 20K context. A 14-file source analysis that cost ~$9.52 in the main session would cost ~$0.50 via subagent. The savings come from context isolation, not from using a cheaper model.

The measurement bug

The cost tracker initially reported $107 for the session. I figured it was a double-count before anyone even dug into it, because we'd been at ~$52 right before. Turned out the cost aggregator was counting compacted session snapshots as separate sessions. Message-ID deduplication fixed it. Actual cost: $64.30.

Lesson: A cost tracker that double-counts is worse than no cost tracker. When the watchdog reported $107, the instinct was to start reasoning about why the session was so expensive. The correct response was to audit the measurement tool first.

Discord threads as session boundaries

One thing I figured out during this session: Discord's built-in threads feature is the right way to manage agent sessions.

I'd been running everything in the main channel, which meant the agent was carrying accumulated context (and cost) from every prior message. Each new topic added to the pile. Starting a new thread gives the agent a clean context, which means lower per-turn cost and better focus.

This isn't something we've enforced in tooling yet. It's an operational practice. But it maps cleanly to the context cost insight: thread boundary = session boundary = context reset.

The point

I woke up at 1 AM with nothing to do. By 10 AM I'd shipped a site redesign, an execution safety agent, a cost monitoring pipeline, and a context management framework. From my phone. In bed.

The reason that's possible is not because the agent is smart. It's because there's a trust model that works. I can see what it wants to do, approve it, and verify the result—all from a Discord thread on my phone. The approval flow is the product.

Building a system you can trust is not primarily a question of capability. It is a question of evidentiary discipline. Gate returns evidence schemas, not "success." The cost pipeline returns freshness labels, not blind numbers. The model doesn't author operational facts. The system provides them.

That discipline is not optional for a system running autonomously. It is the system.

Session cost: $64.30. All of it from a phone.
